## Goal
Capture the current Phase 2 roadmap baseline as private Memora-style planning.
This packet covers milestones 9 through 13 and records which milestone packets
are implementation-grade now versus revisitable draft packets.

## Scope
- keep the milestone sequence as M9 through M13 in the approved order
- preserve M9 as the immediate execution milestone
- write M10 at execution-ready issue depth
- tighten M11 through M13 as draft issue packets
- record expected churn so later revisions stay honest

## Milestone Sequence
1. M9 - Deterministic Project State Model
2. M10 - External Workflow, Codex First
3. M11 - Human Review UI
4. M12 - Existing Repo Understanding MVP
5. M13 - Mobile Contribution MVP

## Writing Depth
- M9: execution-ready now; no broad milestone reframing
- M10: execution-ready next; lock titles, scope, dependencies, acceptance
  criteria, risks, and validation
- M11: tightened draft packet only; revisit after M10
- M12: tightened draft packet only; highest expected change
- M13: tightened draft packet only; high expected change

## Expected Churn
- Lowest change: M9
- Low to moderate change: M10
- Moderate change: M11
- Highest change: M12 and M13

M12 depends on how well M9 through M11 define the deterministic state view,
provenance, and review surfaces. M13 depends on the eventual import shape and non-canonical
intake flow that earlier milestones settle.

## Milestone 9 Packet
Status: execution-ready now

### M9-01 - Define Project State Projection Schema
Type: backend / docs
Summary: define the canonical serialized state view of the current
deterministic project-state output.
Acceptance Criteria:
- `/docs/project-state-view.md` exists
- every field maps to existing code behavior
- no parallel domain model is introduced

### M9-02 - Stabilize Serialized Projection Output
Type: backend
Summary: make the serialized state view stable across identical inputs.
Acceptance Criteria:
- identical input produces byte-identical serialized output
- deterministic ordering is normalized
- unstable fields are excluded from equality-sensitive output

### M9-03 - Add Determinism Tests
Type: backend / test
Summary: prove the repeated state view is reproducible.
Acceptance Criteria:
- the same fixture produces identical output across at least three runs
- tests fail on ordering drift
- cache compatibility remains covered

### M9-04 - Align Projection With Existing Contract Shapes
Type: backend / integration
Summary: confirm the state view fits existing runtime contract shapes before
adding any new top-level operation.
Acceptance Criteria:
- the state view works through existing contract shapes, or
- a documented justification exists for any new operation

### M9-05 - Document Agent Interpretation Rules
Type: docs
Summary: document how agents should interpret canonical and non-canonical
project-state output.
Acceptance Criteria:
- an agent-facing interpretation guide exists
- canonical versus non-canonical meaning is explicit

### M9-06 - Define Projection Boundary and Size Rules
Type: docs / design
Summary: document state-view boundaries before implementing enforcement.
Acceptance Criteria:
- inclusion limits and ordering rules are documented
- no unbounded dump-everything behavior is implied

### M9-07 - Minimal Public vs Non-Canonical Classification
Type: docs
Summary: introduce the smallest classification language agents need.
Acceptance Criteria:
- public repo versus non-canonical planning language is documented
- no new permissions system is implied

## Milestone 10 Packet
Status: execution-ready next
Revision Note: revisit after M9 completes, but keep this packet implementation
grade unless real M9 output forces a change.

### M10-01 - Stabilize MCP Server Entry Surface
Type: integration
Summary: harden `MemoraMcpServer` so it can reliably serve one external client.
Scope:
- normalize request handling across the existing tool surface
- remove debug or partial paths from the operator path used in M10
- add basic failure handling that preserves contract shape
Acceptance Criteria:
- the MCP server serves the existing tool set consistently
- request failures return stable, explainable diagnostics
- no vendor-specific business logic is introduced
Dependencies:
- M9 complete
Risks:
- MCP surface drift from the runtime contract
Validation:
- targeted MCP integration tests for request and error handling

### M10-02 - Expose Deterministic Context Through The Existing Contract
Type: integration / backend
Summary: expose the stabilized state view through MCP using the current shared
contract shape unless a new operation becomes unavoidable.
Scope:
- wire the stabilized state view to the current context retrieval path
- keep the shared contract shape aligned with the runtime-facing docs
- document any contract evolution if M9 proves it necessary
Acceptance Criteria:
- an external client can retrieve the stabilized state view through the shared
  Memora-facing contract
- no duplicate retrieval surface is added without explicit justification
- existing runtime contract tests still pass
Dependencies:
- M9-02
- M9-04
Risks:
- premature contract expansion
Validation:
- runtime-facing compatibility tests plus one MCP retrieval path test

### M10-03 - Implement Proposal Submission Path
Type: backend / integration
Summary: allow the external client to submit reviewable non-canonical
proposals through the shared interaction flow.
Scope:
- use the filesystem-backed interaction service
- persist non-canonical proposal artifacts without touching canonical truth
- preserve lifecycle and approval boundaries
Acceptance Criteria:
- proposal submission creates a non-canonical artifact revision
- canonical artifacts remain unchanged until approval
- validation errors return stable diagnostics
Dependencies:
- M10-02
Risks:
- accidental bleed into canonical storage
Validation:
- API and MCP proposal submission tests

### M10-04 - Implement Outcome Recording Path
Type: backend / integration
Summary: allow recording reviewable outcome artifacts from agent runs.
Scope:
- persist non-canonical outcome artifacts
- keep proposal-to-outcome linkage loose unless M10 evidence proves otherwise
- preserve deterministic storage and lifecycle boundaries
Acceptance Criteria:
- outcome recording writes a reviewable non-canonical artifact
- the operation does not require canonical writes
- missing linkage does not block the first operational flow
Dependencies:
- M10-03
Risks:
- over-coupling proposal and outcome behavior too early
Validation:
- targeted outcome recording tests

### M10-05 - Codex Integration, First Operational Client
Type: integration
Summary: make Codex the first real external client for the M10 golden path.
Scope:
- define the retrieval, planning, proposal, and outcome loop for Codex
- keep the client path grounded in the shared Memora contract
- avoid vendor-specific core logic
Acceptance Criteria:
- Codex can retrieve context without manual copy/paste
- Codex can submit a proposal through the shared contract
- Codex can record an outcome through the same contract surface
Dependencies:
- M10-02
- M10-03
- M10-04
Risks:
- vendor-specific assumptions leaking into the contract
Validation:
- one end-to-end Codex workflow run against the shared surface

### M10-06 - Golden Path Workflow Test
Type: integration / test
Summary: validate the first external workflow end to end.
Scope:
1. retrieve deterministic context
2. produce a plan externally
3. submit a non-canonical proposal
4. record a non-canonical outcome
Acceptance Criteria:
- the full loop works without manual context copying
- the canonical versus non-canonical boundary remains intact
- deterministic retrieval remains explainable across repeated runs
Dependencies:
- M10-01 through M10-05
Risks:
- hidden integration gaps between contract layers
Validation:
- golden path integration test plus manual verification notes

### M10-07 - ChatGPT Read-Only Validation
Type: integration
Summary: verify that a second client can consume the shared state without
taking on mutation behavior yet.
Scope:
- use the shared retrieval shape only
- keep ChatGPT in read and validation mode
- leave write-path behavior for a later milestone
Acceptance Criteria:
- ChatGPT can read the same project-state output through the shared contract
- no mutation-specific behavior is required for the validation path
- the packet stays compatible with future Claude consideration
Dependencies:
- M10-02
- M10-06
Risks:
- pulling multi-client complexity forward too early
Validation:
- documented read-only validation run
