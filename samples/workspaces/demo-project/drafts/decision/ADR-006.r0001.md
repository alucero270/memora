## Context
The term projection was technically defensible, but it suggested a separate
read-model architecture that Memora is intentionally not using. That made the
runtime-facing bundle name harder to understand than the actual behavior.

## Decision
Prefer state view as the primary term for the runtime-facing deterministic read
model. Keep the existing contract shape, but use clearer naming in code and
documentation where possible.

## Alternatives Considered
- keep projection as the primary term everywhere
- rename the runtime contract surface itself
- introduce a second top-level operation to match the old terminology

## Consequences
The read model becomes easier to understand without changing Memora's actual
architecture or introducing a second source of truth.
